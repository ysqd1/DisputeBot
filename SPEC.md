# 反驳生成器 - 需求分析

## 1. 项目概述

**项目名称**: DisputeBot (反驳机器人)
**项目类型**: 本地优先的 Web 应用
**核心功能**: 输入对方的喷言，AI 生成有力的反驳话术

**目标用户**: 计算机相关专业大学生（你），用于学习和娱乐

## 3. 功能需求

### 3.1 核心功能

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 背景场景 | 文本输入：新闻/背景/争论焦点等 | P0 |
| 立场设置 | 设置自己的立场（球迷/玩家等） | P0 |
| 对方定位 | 描述对方身份（结晶/云玩家等） | P0 |
| 激烈程度 | 滑块调节反驳的激烈程度 (1-10) | P0 |
| Temperature | 滑块调节输出随机性 (0-2) | P0 |
| 生成反驳 | 输入对方话语，生成反驳 | P0 |
| 字数控制 | 用户输入目标字数，生成约在该字数附近 | P0 |

### 3.2 信息真实性要求 ⭐⭐⭐ (核心原则)

**这是最重要的原则，必须贯穿整个项目！**

1. **禁止凭空捏造事实**
   - 反驳中的所有事实性陈述必须是真实的
   - 如果不确定某个信息是否正确，必须标注"未经核实"
   - 严禁为了赢而编造数据、成绩、事件

2. **驳倒对方是核心目标**（参考辩论技巧）
   - 指出对方的错误信息（最常见手段）
   - 顺着对方的逻辑推出荒谬结论（归谬法）
   - 指出对方论证的漏洞和逻辑谬误
   - 用事实和数据进行对比反击
   - 不是为了喷而喷，而是以理服人

3. **验证机制**
   - AI 在生成反驳前，应该先识别对方话语中的事实性陈述
   - 如果发现虚假信息，优先指出错误而不是对骂
   - 重要的数据（如比分、奖项、数据）应该逻辑自洽

4. **激烈程度与以理服人的平衡**
   - 激烈程度影响语气和表达方式，但不影响论据真实性
   - 无论激烈程度多高，论据必须站得住脚
   - 温和模式: 摆事实讲道理，逻辑清晰
   - 中等模式: 有理有据，稍带情绪
   - 激烈模式: 阴阳怪气、拉满，但依然以理服人

### 3.3 参数说明

**激烈程度 (Aggression)**: 滑块 1-5
- 1: 极度温和、摆事实讲道理
- 2: 温和但有态度、稍带情绪
- 3: 中等、阴阳怪气、讽刺
- 4: 很强、讽刺挖苦、拉满
- 5: 直接开喷、攻击力极强

**Temperature**: 滑块 0.0-2.0
- 0.0-0.3: 稳定输出，重复性好
- 0.4-0.7: 平衡模式
- 0.8-2.0: 创意模式，可能有意外惊喜

**字数控制 (max_tokens / target_length)**
- 用户输入目标字数（如50字）
- AI 生成约在该字数的回复（prompt引导，无硬性截断）
- max_tokens 固定上限: 3000
- 特殊值: 0 或留空 = 不限制

### 3.4 界面风格

- **简洁实用**: 不追求花哨的动画，注重快速上手和好用
- **响应式**: 支持电脑和手机访问

### 3.5 前端输入框设计

**占位符 (Placeholder)**
- 每个输入框显示浅灰色示例文字
- 用户点击后示例消失，用户输入后正常显示

**示例占位符**:
| 输入框 | 占位符示例 |
|--------|-----------|
| 背景场景 | 如: "足球 - 国家德比皇马0-4巴萨，球迷互喷" |
| 我的立场 | 如: "皇马球迷" |
| 对方定位 | 如: "巴萨球迷结晶" |
| 目标字数 | 如: "200" |
| 对方的话 | 如: "骡子就靠皇马体系，没有本泽马什么都不是" |

**参数说明标签**:
- Temperature 旁边显示: "随机性 (0-2)"
- 激烈程度旁边显示: "激烈程度 (1-10)"
- 字数旁边显示: "字数 (10-2000)"

### 3.6 字数控制

| 范围 | 说明 |
|------|------|
| 下限 10字 | 太短没意义 |
| 上限 2000字 | 较长回复 |
| 0 或留空 | 不限制 |

### 3.7 背景场景（scene）

场景不仅仅是选择领域，而是**完整的背景信息**，包含：

- **领域类型**: 足球/游戏/生活/通用（简化选择）
- **新闻/事件**: 可以粘贴新闻链接或内容
- **争论焦点**: 双方在争什么
- **当前背景**: 最近发生的事、球队状态等

**示例**:
```
足球 - 最近国家德比皇马输了0-4，巴萨球迷在疯狂嘲讽
游戏 - 原神新角色上线，有人说不如王者荣耀
生活 - 同事讨论996，我反对但他们觉得正常
```

## 5. API 设计

### 5.1 生成反驳

**请求**: `POST /api/generate`

```json
{
  "message": "骡子又隐身了，国家德比就这水平？",
  "scene": "足球 - 国家德比皇马客场0-4惨败巴萨，皇马球迷被疯狂嘲讽",
  "my_stance": "皇马球迷",
  "opponent_profile": "巴萨球迷结晶",
  "aggression": 3,
  "temperature": 0.8,
  "target_length": 50,
  "model": "minimax",
  "model_variant": null,
  "api_key": null,
  "base_url": null
}
```

**字段说明**:
- `model`: 供应商ID (minimax/deepseek/kimi/zhipu/custom)
- `model_variant`: 具体模型名称，留空使用默认
- `api_key`: 前端输入则优先使用，否则用 .env 配置
- `base_url`: 仅 custom 时使用，自定义中转站地址

**响应**:
```json
{
  "success": true,
  "reply": "刷出来的？请问骡子连续三年欧冠十六郎是怎么刷的？还体系球员，没有哈白布你什么都不是..."
}
```

### 5.2 可用模型列表

**请求**: `GET /api/models`

**响应**:
```json
{
  "models": [
    {"id": "minimax", "name": "MiniMax", "provider": "MiniMax"},
    {"id": "deepseek", "name": "DeepSeek", "provider": "DeepSeek"},
    {"id": "kimi", "name": "KIMI", "provider": "Moonshot"},
    {"id": "zhipu", "name": "智谱GLM", "provider": "Zhipu"},
    {"id": "custom", "name": "自定义中转站", "provider": "Custom"}
  ],
  "default": "minimax"
}
```

## 6. 学习路线（边做边学）

1. **FastAPI 基础**: 路由、请求、响应
2. **Pydantic 数据验证**: 类型安全
3. **异步编程**: async/await
4. **HTTP 客户端**: 调用外部API
5. **前端交互**: Fetch API + DOM操作

### 6.1 代码生成方式

**分步生成，从小到大**
1. 每生成一小块代码，配一个 `GUIDE-XXX.md` 讲解文件
2. 先让用户理解这块代码是什么、为什么这样写
3. 用户确认理解了，再生成下一块

**每块代码配套文档格式**:
```markdown
# GUIDE-XXX: [功能名称]

## 这块代码做了什么
[简单解释]

## 核心知识点
- 知识点1
- 知识点2

## 代码解读
[逐行或逐段解释]

## 如果我想试试看
[动手实验的建议]
```
